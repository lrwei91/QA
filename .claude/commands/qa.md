# Test Case Generator

分析需求文档、PRD、网页正文、接口说明、现有用例和固定模板，自动生成从环境检查到生成报告的全流程测试用例。

## 路径约定

```bash
# 自动检测：如果当前目录就是工具目录，直接使用；否则从环境变量或默认位置读取
if [ -f "$(pwd)/SKILL.md" ]; then
  TOOL_DIR="$(pwd)"
else
  TOOL_DIR="${TEST_CASE_GENERATOR_DIR:-$HOME/.claude/test-case-generator}"
fi
```

### 步骤 3：提取关键要素（自动）

至少提取以下信息：

- 功能模块和子模块

- 输出项、返回结果、错误码或页面反馈

- 边界条件、异常条件
- 是否存在已有测试用例需要避重补充
- 是否存在指定模板、固定区、表头行或导出要求

> - 核心功能点：[列表]
>
> 请选择要执行的操作：
> 1. 生成冒烟用例
> 2. 生成完整测试用例
> 3. 补充已有用例（增量去重）
> 4. 生成多语言校验 JSON
> 5. 仅分析需求，不生成用例"

### 步骤 7：判定平台拆分（自动）

`平台` 字段只使用以下业务统一名称：

- `客户端`：页面展示、控件交互、文案提示、跳转、渲染、表单输入、按钮点击、弹窗、列表、禁用态、前端校验、交互反馈、视觉状态
- `账服`：接口入参/出参、服务端校验、业务处理、状态变更、写库/查库、缓存、消息、异步任务、错误码、幂等、权限、风控、限流

**若同时涉及前后端**：不要强行合并成一个平台，拆分为两条用例：

- 页面交互与展示 → `客户端`
- 接口校验、处理逻辑、返回结果和数据落库 → `账服`

### 步骤 8：设计测试用例（自动）

先应用通用测试设计策略，再按项目模块补关联场景。

#### 8.0) 冒烟用例筛选规则（仅当用户选择"生成冒烟用例"时）

当用户选择生成冒烟用例时，在完整测试用例设计完成后，按以下规则筛选：

**保留标准（只保留以下类型）：**

- ✅ 正向主流程：核心功能的最基本成功路径
- ✅ 关键权限校验：没有通过则后续无法进行的权限检查
- ✅ 关键状态流转：影响业务核心状态变更的验证
- ✅ 必填项校验：最基础的必填字段检查

**剔除标准：**

- ❌ 边界值测试（除非边界是核心业务规则）
- ❌ 异常/反向测试（除必填校验外）
- ❌ 次要功能点的验证
- ❌ 日志、通知、统计口径等辅助功能
- ❌ P1/P2/P3 优先级的用例

**筛选步骤：**

1. 先按完整流程设计完整测试用例
2. 筛选 P0 优先级用例（正向主流程 + 关键校验）
3. 确保每个核心功能点至少有 1 条冒烟用例
4. 控制总量在完整用例的 10%-20%
5. 输出冒烟用例表格（格式同完整用例）

---

#### 8.1) 基础覆盖面检查

每个核心功能点至少从以下维度判断是否需要覆盖：

- 正向主流程
- 必填 / 合法性校验
- 异常输入或非法操作
- 边界值
- 状态流转
- 权限或角色差异
- 数据结果或页面反馈一致性
- 前后端联动拆分

对后台管理系统额外检查：

- 日志
- 通知
- 统计口径
- 数据一致性
- 导出导入
- 查询筛选
- 幂等性

#### 8.2) 典型设计方法

默认综合使用以下方法：

- 场景法：按真实用户路径组织主流程
- 等价类：减少同质化冗余
- 边界值：覆盖上下限、临界点、空值、未传
- 状态迁移：覆盖合法流转和非法流转
- 异常分析：覆盖参数非法、越权、重复操作、中断、失败回退

#### 8.3) 必须补充关联用例的信号

出现以下任一信号时，必须补充关联用例：

- 权限、角色、可见范围、数据范围
- 登录、会话、账号状态、组织归属
- 新增、编辑、删除、审批、发布、配置变更
- 状态流转、审批结果、异步通知
- 商品、订单、退款、营销状态变化
- 统计口径、看板、日志、审计、消息触达

---

### 步骤 9：输出用例（自动）

若用户未指定模板，严格使用 `references/output-template.md` 中的默认列结构：

| 序号 | 平台 | 模块 | 功能点 | 前置条件（测试点） | 操作步骤 | 预期结果 | 测试结果 | 备注 |
|------|------|------|--------|-------------------|----------|----------|----------|------|

**字段规范：**

- `序号`：连续递增
- `平台`：仅使用 `账服` 或 `客户端`
- `模块`：写功能所属模块，不要过于笼统
- `功能点`：写被验证的具体点，不要只写"功能测试"
- `前置条件（测试点）`：写执行前必须满足的状态、角色、数据、配置
- `操作步骤`：写清操作顺序，多行时使用 `1、2、3、` 编号换行
- `预期结果`：与步骤对应，写具体到页面展示什么、接口返回什么、是否提示错误、状态是否变更
- `测试结果`：默认留空
- `备注`：至少标记用例类型，推荐同时带优先级，例如 `【功能测试】【P0】`

**不要只写：**

- 功能正常
- 提示正确
- 返回成功

### 步骤 10：多语言 JSON（条件触发）

当需求、附件或对话中明确出现多语言文案，并且固定语言集合都已提供时，额外输出一份多语言校验 JSON。

**固定语言集合：**

- `en-us`
- `id-id`
- `pt-pt`
- `es-es`
- `bn-bn`
- `tr-tr`
- `fp-fp`

**生成规则：**

- 只在全部语言值完整时生成 JSON
- 若缺语言，不自动补翻译，不生成 JSON，只输出缺失语言清单
- 若语言完整但验证配置缺失，生成草稿 JSON（`url=""`，`preScriptPath=""` 等）

```json
{
  "name": "<功能名称>",
  "url": "<验证页面 URL>",
  "preScriptPath": "<前置脚本路径>",
  "languages": {
    "en-us": { "key": "value" },
    "id-id": { "key": "value" },
    ...
  },
  "options": {
    "matchRule": "normalized-exact",
    "captureRegion": { "x": 0, "y": 0, "width": 0, "height": 0 }
  }
}
```

### 步骤 11：处理已有用例补充（条件触发）

当用户提供已有用例并要求"继续补充""只补新增""避免重复"时：

- 只输出新增测试用例
- 不改写已有用例
- 不重排已有用例内容
- 旧用例仅用于判断覆盖范围和避重
- 若当前工作区存在索引文件，优先结合索引命中的历史文件一起判断覆盖缺口和重复
- 优先补齐遗漏的功能、异常、边界、状态和权限场景
- 如果没有真正新增的场景，返回空结果，不要重复旧用例

### 步骤 12：导出与保存（用户确认后执行）

**默认行为**：在对话中直接输出结构化测试用例；若语言完整，也可同时输出多语言 JSON 内容。

**冒烟用例导出约定**：

- 若用户选择"生成冒烟用例"，输出文件名默认带 `_smoke` 标识
- 例如：`测试用例_登录模块_smoke_20260330.md` 或 `测试用例_登录模块_smoke_20260330.xlsx`
- 冒烟用例总量控制在完整用例的 10%-20%

**只有在用户明确要求保存、导出或写入某路径时**，才执行落盘：

```bash
# 若用户指定完整路径，直接写入指定文件
# 若用户只指定目录，默认文件名：测试用例_<模块或主题>_<yyyymmdd>.md
# 若用户希望落盘但未指定路径，默认写入：testcases/generated/

# 检查仓位是否存在
if [ ! -d "testcases" ]; then
  mkdir -p testcases/generated testcases/i18n
  echo "CREATED_SLOTS"
fi
```

**Excel 导出要求：**

- 列顺序与目标模板保持一致
- 多行字段必须写入真实换行符
- `前置条件（测试点）`、`操作步骤`、`预期结果` 必须开启自动换行（wrap text）
- 固定区不得被覆盖、删除或上移
- 表头行位置不得改变
- 统计公式应覆盖当前数据区
- 默认留空字段（测试平台、系统&版本、文档编写人、参考档、测试日期、最后更新）若无用户输入，必须保持为空

**索引维护：**

若输出写入工作区产物仓位，写完后同步新增或更新对应索引条目：

```bash
# 优先使用脚本自动生成或更新索引
python3 scripts/upsert_testcase_index.py --file "<生成的文件路径>" 2>&1

# 多语言 JSON 合法性校验
python3 scripts/validate_i18n_json.py --file "<多语言 JSON 路径>" 2>&1
```

### 步骤 13：质量自检（自动）

输出前至少检查以下问题：

**通用检查（所有用例）：**

- 每个核心功能点是否至少有 1 条正向用例
- 关键校验是否有异常用例
- 有边界约束的字段是否覆盖边界值
- 状态流转是否覆盖合法与非法路径
- 是否遗漏角色 / 权限差异
- 是否存在空模块、空功能点或空预期结果
- 是否把前后端混成一条
- 是否存在明显重复或低价值重复
- 备注是否标记了用例类型
- 是否把说明性文字混入了结果体
- 若导出 Excel，多行字段是否已开启自动换行
- 若使用模板，固定区、表头行、统计区、默认留空字段是否都符合要求
- 若需求含多语言内容，是否检查了固定语言集合是否完整
- 若生成多语言 JSON，是否符合固定 schema，并已同步写入多语言索引

**冒烟用例额外检查：**

- 是否只保留了 P0 优先级用例
- 每个核心功能点是否至少有 1 条冒烟用例
- 是否已剔除边界值、异常测试等非核心用例
- 总量是否控制在完整用例的 10%-20%

---

### 步骤 14：维护模式（仅用户明确要求时）

**默认只输出**：

- 测试用例结果
- 索引增量建议
- 模块文档增量建议

**只有在用户明确要求更新 skill 时**，才执行以下动作：

1. 先根据 `references/update-governance.md` 判断要改哪些资源
2. 修改 `references/module-index.json` 与对应领域文件
3. 必要时同步更新 `references/system-map.md`、`references/platform-rules.md` 或 `references/output-template.md`
4. 运行 `python3 scripts/validate_index.py references/module-index.json` 校验索引
5. 若用户要求交付安装包，重新打包整个 skill

## 错误速查

| 错误 | 自动处理 | 需用户操作 |
|------|---------|-----------|
| 模块匹配歧义 | 展示候选模块列表 | 用 AskUserQuestion 确认主模块 |
| 需求类型不明确 | 基于现有内容做最合理匹配 | 补充更多需求细节 |
| 模板固定区识别失败 | 展示识别结果请用户确认 | 手动指出表头行位置 |
| 多语言不完整 | 输出缺失语言清单 | 补充缺失语言文案 |
| 历史用例索引不存在 | 自动创建仓位和索引骨架 | 若无需保存则忽略 |
| 平台拆分模糊 | 根据核心结果归属判断 | 若仍模糊则询问用户 |
| 已有用例补充无新增 | 返回空结果 | 检查是否确实有遗漏场景 |
| Excel 导出依赖缺失 | 自动 `pip install openpyxl pandas` | — |
| 索引校验失败 | 展示错误详情 | 按错误提示修复后重试 |
| 冒烟用例筛选后数量过少 | 提示用户可能需求核心功能点较少 | 确认是否需要扩展为完整用例 |
| 冒烟用例与完整用例比例异常 | 提示用户当前比例（正常 10%-20%） | 确认是否调整筛选标准 |
