# 用途：客户端/账服平台划分规则 - 定义「账服」和「客户端」的判断信号，指导测试用例的「平台」字段归类

# 平台划分规则

`平台` 字段只填写以下两种业务侧统一名称：

- `账服`
- `客户端`

其中：

- `账服` 表示后台、服务端、后端、接口服务、业务服务等服务端能力
- `客户端` 表示 web 前端、前端页面、浏览器端、h5 页面等前端能力

## 1. 归类到 `账服` 的典型信号

出现以下内容时，优先归类为 `账服`：

- 接口、api、请求方法、url、路由、网关
- 鉴权、授权、登录态、token、session、签名、权限校验
- 数据库、表、写库、落库、查询、事务、缓存、消息队列
- 任务、定时任务、异步处理、回调、重试、幂等
- 返回码、错误码、返回报文、字段校验、服务异常
- 服务逻辑、业务规则校验、状态流转、风控、限流
- 管理后台配置、后台开关、后台策略、后台任务处理

如果描述里出现“账服”“后台”“服务端”“后端接口”“服务处理”等说法，默认判为 `账服`。

## 2. 归类到 `客户端` 的典型信号

出现以下内容时，优先归类为 `客户端`：

- web 页面、前端页面、浏览器页面、h5 页面
- 页面展示、ui 展示、文案展示、样式展示、列表展示
- 表单输入、按钮点击、弹窗、下拉框、勾选框、搜索框
- 页面跳转、路由跳转、刷新、加载、渲染、可见性
- 交互反馈、前端校验、toast、提示语、禁用态、置灰
- 兼容性、浏览器行为、响应式布局、前端交互流程

如果描述里出现“客户端”“前端”“web 前端”“页面端”“浏览器端”等说法，默认判为 `客户端`。

## 3. 冲突与拆分规则

如果同一条需求同时包含前后端内容，不要强行合并成一个平台，应该拆成两条或多条测试用例：

- 页面发起操作、按钮点击、前端展示与交互，归为 `客户端`
- 请求入参校验、服务处理、写库、返回结果、状态变化，归为 `账服`

例如：

- “点击领取按钮后，页面展示成功提示” 归为 `客户端`
- “领取接口校验资格并写入领取记录” 归为 `账服`

## 4. 优先判定原则

当一句描述同时含有页面和接口词汇时，按“测试关注点”决定归类：

- 关注用户看到什么、怎么点、怎么跳、怎么提示，用 `客户端`
- 关注服务怎么校验、怎么处理、怎么返回、怎么落库，用 `账服`

如果仍然模糊，优先看预期结果：

- 预期结果偏页面表现，归 `客户端`
- 预期结果偏接口返回或数据结果，归 `账服`

## 5. 与索引联动的使用方式

读取 `index-rules/module-index.json` 时，同时使用：

- `platform_scope`：判断该模块是否可能同时覆盖前后端
- `client_signals`：判断更偏页面交互的命中语义
- `server_signals`：判断更偏服务处理的命中语义

如果同一模块同时命中 `client_signals` 与 `server_signals`，通常应该输出两侧用例，而不是只给一个平台。
