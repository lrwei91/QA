# Test Case Taxonomy

始终先使用通用测试设计策略思考覆盖范围，再按模块索引扩展后台管理系统的关联用例。

## 1. 正向测试

为每个核心功能点或接口至少设计 1 条正常路径用例，明确成功条件，例如：

- 页面正常展示
- 操作成功
- 状态正确流转
- 返回码正确
- 关键字段正确入库或返回

## 2. 反向 / 异常测试

针对每个可校验条件，至少考虑一类无效输入或异常操作，例如：

- 必填缺失
- 格式错误
- 类型错误
- 参数非法
- 越权访问
- 重复提交
- 状态不允许

预期结果必须具体，不要只写“提示错误”，而要写清：

- 错误提示
- 错误码
- 不写库
- 不变更状态
- 不触发后续动作

## 3. 边界值测试

对所有有长度、范围、数量、次数、上限/下限约束的字段或条件设计边界用例，至少考虑：

- 最小有效值
- 最大有效值
- 刚好等于边界
- 刚好越过边界
- 空值、空串、null、未传

## 4. 等价类划分

在输入域较大的情况下，用等价类减少冗余：

- 为有效等价类挑选代表值
- 为每种无效类型挑选代表值
- 边界附近的值可以同时承担边界测试和等价类代表

## 5. 状态与流程测试

如果需求涉及状态机、审批流、多步骤业务流程、角色切换或权限变化：

- 列出主要状态和允许迁移
- 设计主流程
- 设计回退 / 中断流程
- 设计非法状态下执行操作的用例
- 覆盖不同角色的可见性和操作范围

## 6. 场景法

对功能类需求，优先按真实用户场景串联多个功能点：

- 提炼 2 到 3 个典型业务场景
- 每个场景至少覆盖 1 条主流程
- 为高风险场景增加异常或边界变体

## 7. 后台管理系统补充维度

当需求属于后台管理系统时，额外检查下列高风险维度：

- 权限与数据范围
- 操作日志与审批轨迹
- 消息通知与触达失败重试
- 统计口径、看板刷新、筛选维度
- 导入 / 导出、批量操作、异步任务
- 幂等性、重复提交、并发操作
- 前后端展示一致性和数据回显一致性

## 8. 优先级建议

若用户未指定其他规则，可按下列方式标记优先级：

- `p0`：核心主流程、关键权限校验、关键状态流转、关键错误处理
- `p1`：常见异常、重要边界、统计口径、日志与通知联动
- `p2`：次要边界、低频异常、易恢复的提示类问题
- `p3`：体验优化类、低风险兼容性类补充检查

优先级可与用例类型一起写入 `备注`，例如：

- `【功能测试】【p0】`
- `【异常测试】【p1】`
- `【边界测试】【p1】`
